1 · Dockerfile built

Single-stage build using python:3.12-slim-bookworm. No system packages needed (psycopg2-binary ships libpq). Runs as non-root pipeline user (UID 1000).

Key design decisions

  • No apt-get — avoids Walmart DNS firewall issues with debian repos
  • Walmart PyPI mirror by default — pypi.ci.artifacts.walmart.com overridable via --build-arg PYPI_INDEX_URL=...
  • Layer caching — requirements.txt copied first; source-only edits skip pip reinstall
  • ENTRYPOINT = python -m — override the CMD to run any module (e.g., scripts.generate_mock_data)
  • Non-root user — CIS best practice, required by some k8s policies

2 · .dockerignore ready

Excludes .venv/, data/, docs/, .git/, tests, and IDE junk. Keeps build context tiny (~200KB instead of ~500MB).

3 · gcp/build_and_push.sh ready

Idempotent script that:

  1. Creates the Artifact Registry repo if missing
  2. Configures docker auth for REGION-docker.pkg.dev
  3. Builds for linux/amd64 (cloud-required)
  4. Tags with both :latest and :<git-sha> for traceability
  5. Pushes both tags

4 · airflow/dags/fraud_pipeline_dag.py ready

Cloud Composer 2/3 DAG using KubernetesPodOperator. Manual trigger only (schedule=None) — no surprise costs.

DAG flow

generate_mock_data  →  run_fraud_pipeline  →  summarize_alerts
   (your image)         (your image)         (gcloud sdk)

All three tasks run as ephemeral K8s pods. is_delete_operator_pod=True ensures pods are GC'd after task completion — zero zombie cost.

Triggered via Airflow UI or CLI:

airflow dags trigger fraud_detection_pipeline \
  --conf '{"num_wms": 5000, "num_invoices": 5000}'

Build arm64 image (local testing, fast) working

Native build on M-series Macs — takes ~3 min. Useful for shape testing, not for cloud.

cd ~/workspace/supply-chain-fraud-detection
docker build --load -t fraud-detection-pipeline:arm64 .
Beam quirk on M1 + podman Beam's arm64 wheels can trip a SIGILL inside podman's hypervisor emulation. The image itself is fine — it just can't run end-to-end under podman on M1. Use it for layer-testing, then build amd64 for actual runs.

Build amd64 image (cloud-ready, slow on M1) ~15 min on M1

x86_64 build on Apple Silicon uses QEMU emulation — slow but produces what GCP needs.

# Foreground (will block for ~15 min)
docker build --platform=linux/amd64 --load -t fraud-detection-pipeline:amd64 .

# Or background (recommended) — check progress with: tail -f /tmp/amd64_build.log
nohup docker build --platform=linux/amd64 --load -t fraud-detection-pipeline:amd64 . \
  > /tmp/amd64_build.log 2>&1 &
Faster alternative: Cloud Build Once GCP is unblocked, use gcloud builds submit --tag REGION-docker.pkg.dev/PROJECT/REPO/fraud-pipeline:latest — builds happen on Google's native x86_64 infra. Takes ~3 min instead of 15.

Set up Cloud Composer (Airflow on GCP) needs project

  1. Create Composer environment (one-time, ~25 min): 
    gcloud composer environments create fraud-airflow \
  --location=us-central1 \
  --image-version=composer-3-airflow-2.10.5 \
  --environment-size=small
  2. Upload the DAG file: 
    gcloud composer environments storage dags import \
  --environment=fraud-airflow \
  --location=us-central1 \
  --source=airflow/dags/fraud_pipeline_dag.py
  3. Set Airflow Variables (in Composer UI → Admin → Variables):
    KeyValue
    gcp_project_idyour-project-id
    fraud_image_urius-central1-docker.pkg.dev/.../fraud-detection-pipeline:latest
    bq_datasetfraud_detection
    gcs_bucketyour-project-fraud-pipeline
Composer cost reality check Composer "small" env runs ~$100/month idle (it's a managed Airflow cluster — always on). Only use this if you actually have ongoing scheduled work. For purely manual triggers, use Cloud Run Jobs instead (~$0 idle).

Trigger the DAG after Composer is up

Via Airflow UI:

  1. Open Composer environment → "Open Airflow UI" button
  2. Find fraud_detection_pipeline DAG
  3. Toggle it ON (it'll never auto-run since schedule=None)
  4. Click "Trigger DAG" → optionally pass config: {"num_wms": 5000, "num_invoices": 5000}

Watch each task's pod logs in real-time. Query BigQuery to see results.

Future: swap to PubSub streaming

When you're ready for true streaming (Phase 3):

  1. Change pipeline/gcp_batch_main.py read step from ReadFromText → ReadFromPubSub
  2. Change runner from DirectRunner → DataflowRunner with streaming flag
  3. Airflow's role shifts: instead of running the pipeline, it monitors the long-running Dataflow streaming job and restarts on failure
  4. Use DataflowStartFlexTemplateOperator in your DAG instead of KubernetesPodOperator

The Dockerfile + image you have now can serve as the publisher (mock data into PubSub) while a separate Dataflow job consumes.
