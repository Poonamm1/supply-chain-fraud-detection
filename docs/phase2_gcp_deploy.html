Why batch > streaming for your use case

You said: "I don't want any job piled up data and I end up paying GCP." Smart instinct. Here's the brutal truth about Dataflow streaming costs:

ModeAlways-on?Min monthly costGood for
Streaming (Pub/Sub → Dataflow → BQ)YES, 24/7~$50-200/mo
(idle worker + streaming engine)		Real prod with non-stop traffic
Batch on-demand ✅ (what we're building)NO, only when triggered$0/mo idle
~$0.05-0.10 per run		Learning, dev, periodic detection
Design Decision Phase 2 uses batch mode triggered manually. The Dataflow job spins up, reads your GCS file, writes to BigQuery, and shuts down completely. No idle workers. No streaming engine. No surprise bills.

You can graduate to streaming later when you have real Pub/Sub traffic and a real budget. For learning + occasional runs, batch is the right answer.

5 · Run the pipeline on demand the fun part

This is the script you'll run every time you want fraud detection to happen. Nothing automatic — only when you execute this.

# Defaults: 1000 WMS + 1000 ERP events. Override if you want.
./gcp/trigger_pipeline.sh

# Or generate more / fewer events:
NUM_WMS=5000 NUM_INVOICES=5000 ./gcp/trigger_pipeline.sh

What this does (in order)

  1. Generates mock data locally via scripts/generate_mock_data.py — free, offline
  2. Uploads JSONL files to GCS at gs://BUCKET/input/RUN_ID/
  3. Submits a Dataflow BATCH job with --max_num_workers=2 --machine_type=e2-small
  4. Dataflow processes everything and EXITS — worker scales to zero

While it's running

Watch the job in real-time at:

https://console.cloud.google.com/dataflow/jobs?project=${PROJECT_ID}

You'll see the DAG visually — same graph as Phase 1, just on Google's infra. Each transform shows live throughput + worker count.

First run is slow (~5 min) Dataflow has to upload your code package to GCS and spin up worker VMs from scratch. Subsequent runs are faster (~2-3 min) thanks to cached images.

Teardown — stop all costs when done experimenting

When you're done playing for the day/week/forever:

./gcp/teardown.sh

This kills any active Dataflow jobs, deletes the BigQuery dataset, and removes the GCS bucket. Recurring cost drops to $0.

Keeps the project alive Teardown does NOT delete your GCP project. So you can re-run setup.sh later without losing your project ID, IAM grants, or budget alert.

If you want to delete the ENTIRE project (true nuclear):

gcloud projects delete ${PROJECT_ID}

Troubleshooting

"Quota exceeded" on Dataflow

New GCP projects have low default quotas. Request a quota increase for "Dataflow CPU quota" in your region. Usually approved in < 1 hour.

"Permission denied" on BigQuery

Re-run ./gcp/setup.sh — the IAM bindings are idempotent and will re-apply. Wait 60 sec for propagation.

Job runs but writes nothing to BigQuery

Check the Dataflow job logs in the Console. Most common cause: the service account can't write to the dataset. Verify with:

gcloud projects get-iam-policy ${PROJECT_ID} \
  --flatten="bindings[].members" \
  --filter="bindings.members:fraud-pipeline-sa*" \
  --format="table(bindings.role)"

"Module 'pipeline' not found" on workers

Dataflow workers couldn't import your local code. Make sure setup.py exists in the project root (it does — Sweety put one there) and you ran with --setup_file=./setup.py (the trigger script does this).

Bill is higher than expected

Check the billing dashboard's "Cost breakdown by SKU". The usual culprit is a forgotten streaming job. Run teardown.sh immediately and inspect Dataflow → Jobs → filter by "Running".

Beyond Phase 2 (when you're ready for real-time)

Once batch is comfortable and you have a real production need:

  • Phase 3 — True streaming: swap ReadFromText → ReadFromPubSub. Switch to streaming mode. Budget ~$100/mo for a minimum streaming engine setup.
  • Cloud Scheduler: if you want batch to run on a cron schedule (e.g., every 4 hours), wrap trigger_pipeline.sh in a Cloud Run Job and schedule it.
  • Flex Templates: package the pipeline as a one-click "Run from console" template for non-engineers.
  • Looker Studio dashboard: connect to gold_fraud_alerts for a free real-time fraud dashboard.
  • Alerting: Cloud Functions watching gold_fraud_alerts via Pub/Sub trigger → email/Slack on CRITICAL.
Don't skip steps Master batch first. It teaches you Beam-on-GCP, BigQuery partitioning, IAM, billing — all without burning money. Phase 3 streaming is way easier once Phase 2 is muscle memory.
